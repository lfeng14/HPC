- **四种迁移方式（和 Neon 一一对应）**
   - **编译器自动向量化**
     编译器直接帮你生成 SVE 指令，不用改太多代码。
   - **Intrinsics 内置函数**
     用 SVE 专用 intrinsic，直接控制指令，比汇编简单、比自动优化可控。
   - **优化数学库**
     直接用支持 SVE 的库（如 Arm Performance Libraries），几乎不用改业务代码。
   - **手写汇编**
     适合资深开发者，能极致调优，但难度最高、移植性差。

- 使用restrict
  - 1. **`restrict` 是什么**
     是 C99 引入的关键字（C++ 常用 `__restrict__`），用来**修饰指针**。
  - 2. **它告诉编译器什么**
     被 `restrict` 修饰的指针，**在它有效期间，不会和别的指针指向同一块内存**（即**没有指针别名 / no alias**）。
  - 3. **对向量化（Neon/SVE 自动优化）的意义**
     编译器一旦确认**没有别名冲突**，就敢大胆优化：
     - 认为循环的每一次迭代**互相独立**
     - 可以安全地**并行执行**
     - 更容易开启 **自动向量化（vectorization）**，生成 Neon/SVE 指令。
  - **`restrict` 用来承诺“这个指针不会和别的指针重叠”，让编译器放心把循环改成并行/向量指令。**


- **算法越复杂，编译器自动向量化（用 Neon/SVE 并行指令优化）的成功率就越低。**

- 下面这几类循环**最难/无法被自动向量化**：
  - 1. **循环迭代之间有数据依赖**：这一轮计算要用到上一轮的结果，没法并行跑。
  - 2. **循环里有 break**：提前退出会让总次数不确定，编译器不敢乱并行。
  - 3. **条件判断太复杂**：编译器分析不出哪些操作能安全并行。

- 简单说：**循环越“干净、规律、无依赖、无提前退出”，越容易自动并行；越乱越难。**


- 1. **SVE 汇编怎么上手**
   - 懂 Neon 汇编的人，可以对照 **ACLE for SVE** 指令集和 **PCS** 调用规范，把 Neon 汇编改成 SVE 汇编。

- 2. **SVE 汇编性能优化要点**
   - 尽早加载寄存器，减少流水线停顿
   - 尽量把值存在寄存器里
   - 避免上一条指令的目标寄存器当下一条的源寄存器
   - 少把寄存器数据溢出到栈
   - 写 **VLA（向量长度无关）** 汇编，不硬编码循环步长
   - 多流水线架构下循环展开
   - 用预取指令提升缓存效率

- 3. **内联汇编（inline asm）**
   - 在 C/C++ 里直接嵌汇编，不用写完整 .s 文件
   - 要求熟悉 SVE：**向量长度无关、谓词、WHILE 指令**等
   - 优点：
     - 编译器帮你处理 PCS、栈、寄存器保存
     - 编译器能看到汇编行为，更好优化
     - 支持函数内联
     - 可直接用 C 里的常量（如结构体偏移）

- 4. **官方建议**
   - 汇编能做到**极致性能控制**，但难度高
   - 只推荐**资深汇编开发者**使用，普通场景编译器优化更高效
